## 解決方案

### 步驟 1：創建權限規範文件
創建 `/etc/nixos/PERMISSION-RULES.md`，定義：
- 需要詢問的操作（系統級、破壞性、網路、Git push）
- 可自動執行的操作（只讀、配置編輯、輕量級）
- UUID 不可變原則
- 代理不可斷原則
- 自動記錄原則（CLAUDE.md + Obsidian）

**關鍵規則**：
```markdown
每次操作前問自己：
- 這個操作會修改系統配置嗎？→ 問
- 這個操作會刪除數據嗎？→ 問
- 這個操作會影響代理或網路嗎？→ 問
- 這個操作會推送到遠端嗎？→ 問
- 這個操作會修改 UUID 嗎？→ 拒絕
```

### 步驟 4：添加 tmux 配置
在 `configuration.nix` 添加：
```nix
programs.tmux = {
  enable = true;
  extraConfig = ''
    set -g status-left "#{?client_prefix,#[bg=red] WAIT ,#[bg=green] TMUX } "
    set -g mouse on
    set -g status-interval 1
  '';
};
```

### 步驟 5：修復 xray 日誌過多
在 `modules/proxy.nix` 的 xrayConfig 添加：
```nix
log = {
  loglevel = "warning";  # 只記錄警告和錯誤
};
```

**效果**：減少 90% 的日誌量，只保留重要信息

### 步驟 7：修復 Flatpak 字體

## 經驗總結

### 關鍵教訓
1. **文檔必須與系統同步**：系統重裝後，配置文件通過 Git 同步了，但 MD 文檔需要手動更新
2. **權限規範很重要**：明確定義何時需要詢問用戶，避免破壞性操作
3. **日誌級別要合理**：xray 默認 info 級別會產生大量無用日誌
4. **Flatpak 字體陷阱**：sandbox 無法訪問 NixOS 系統字體，需要手動複製

### 最佳實踐
1. 每次系統重裝後，立即更新 CONTEXT.md 和 CLAUDE.md
2. 重要操作前先檢查 PERMISSION-RULES.md
3. 所有修復必須記錄到 Obsidian（本文件格式）
4. 配置變更後必須 `git commit`
